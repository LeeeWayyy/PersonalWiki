# Vault Schema

The rulebook for the LLM-maintained wiki. Every ingest follows this.
If the ingest prompt can't follow these rules, fix the prompt.

## 0. Wiki languages — source-language wins

**There is no fixed canonical language.** Each new page is written in
the language of the source that introduced it. The page then keeps
that language forever; future sources update it in that same
language regardless of their own.

**Working-language pair:** `English` + `中文` (Chinese). These are
the two languages the vault routinely handles. `aliases:` on every
page includes both forms when known.

Rules:

1. **New page from a source** → page language = source language.
   H1, prose, and wikilinks on the page all use that language.
2. **Updating an existing page** (found by alias lookup) → new
   content is written in the existing page's language, not the
   source's. This keeps each page internally monolingual and
   coherent.
3. **Aliases** must include both English and Chinese forms (at
   minimum) when known. If the secondary-language form is genuinely
   unknown (rare transliteration, coined term), omit rather than
   invent.
4. **Entity-page dedup** — before creating a new page for an entity,
   look through existing pages' `aliases:` for a match. If a match
   exists, update that page instead.

Source-language fidelity is further preserved via:
- Parenthetical first-mention when the page's language differs from
  the source: `线粒体 (Mitochondria)` on a Chinese page that also
  cites English sources.
- Quoted spans in the source language when wording matters.

## 1. Page types

Two content types:

- **`Entity`** — a person, project, concept, or anything singular you
  want a dedicated page for. Lives in `wiki/entities/`.
- **`Topic`** — a synthesis page spanning multiple entities or
  sources. Lives in `wiki/topics/`.

Plus two tool-owned auxiliary types:

- **`index`** — Map-of-Content (MOC) page generated by
  `scripts/generate-mocs.py`. One MOC per `tags:` value. Lives in
  `wiki/_index/`. **Not subject to §2** (no `page_id`, no
  `sources`, no `aliases`, no `last_ingested` required). MOC
  editing rules:
  - `human-zone` is preserved verbatim across regenerations — the
    safe place for human commentary.
  - `moc-zone` is regenerated on every run — DO NOT hand-edit;
    edits will be erased.
  - Frontmatter is canonical and overwritten — DO NOT add custom
    fields; they will be erased on the next regeneration. Put
    metadata in `human-zone` prose instead.

- **`map`** — book/source mind-map page generated by
  `scripts/generate-mindmap.py`. One map per source `source_id` (a
  multi-file work may later be unified via a `map_group:` sidecar
  field). Lives in `wiki/_maps/`. **Not subject to §2** (no
  `page_id`, no `sources`, no `aliases`, no `last_ingested`
  required). Same editing rules as `index`:
  - `human-zone` is preserved verbatim across regenerations.
  - `map-zone` (a Mermaid `graph` + a `[[wikilink]]` outline) is
    regenerated on every run — DO NOT hand-edit.
  - Frontmatter is canonical and overwritten — DO NOT add custom
    fields.
  - The map is a **derived view**, never a source: it **is never cited as
    evidence** by other pages (§6). Its nodes/spine MAY carry `[src:<id>#…]`
    **anchors to the sources they summarize** (the LLM argument map does — for
    auditability); the rule is "a map is never *cited as* a source," not "a map
    carries no `[src:]` anchors." It cannot drift independently of the pages /
    sources it summarizes.

- **`reading`** — Japanese language-learning reader artifacts generated by
  `scripts/generate-language-pages.py` under the **isolated `content/lang/`
  subtree** (the `--profile lang` ingest path; never mixed into the wiki).
  Each source emits `_reading/<slug>.html` for standalone local reading and
  `_reading/<slug>.reading.json` for the website reader / word bank. The JSON
  schema is `reading/2` and is the authoritative language-reader contract.
  **Not subject to §2** (no `page_id`, no wiki `sources`, no `aliases`, no
  `last_ingested`). Editing rules:
  - `_reading/*.html` and `_reading/*.reading.json` are regenerated on every
    run — DO NOT hand-edit; edits will be erased.
  - These are **derived views**, never sources (§6): never cited as evidence
    by other pages. The HTML carries `[src:<id>]` and `[src:<id>#<chapter>]`
    anchors to the source it summarizes (for auditability), and cannot drift
    independently of that source.
  - Legacy `_study`, `_vocab`, and `_grammar` markdown artifacts are retired
    and are not read by the app. Re-ingest old language sources with
    `--profile lang` to generate current `reading/2` JSON.

`reading/2` shape emitted by `scripts/generate-language-pages.py`:

```json
{
  "schema": "reading/2",
  "source_id": "01K...",
  "title": "Display title",
  "lang": "ja",
  "prompt_version": "v4",
  "chapters": [
    {
      "chapter": "第1章",
      "paragraphs": [
        {
          "sentences": [
            {
              "jp": "原文。",
              "en": "Target-language translation.",
              "tokens": [
                {
                  "t": "原文",
                  "rt": "げんぶん",
                  "w": "原文",
                  "m": "gloss",
                  "n": "optional note",
                  "pos": "名詞",
                  "key": "原文\u001fゲンブン\u001f名詞",
                  "new": true
                }
              ],
              "grammar": [
                {
                  "pattern": "〜ている",
                  "explanation": "Short learner-facing explanation.",
                  "example_jp": "原文の例。",
                  "s": 1
                }
              ]
            }
          ]
        }
      ],
      "grammar": [
        {
          "pattern": "〜ている",
          "explanation": "Short learner-facing explanation.",
          "example_jp": "原文の例。",
          "s": 1
        }
      ]
    }
  ]
}
```

Required fields:
- Top level: `schema` exactly `reading/2`, `source_id`, `title`, `lang`,
  `prompt_version`, and ordered `chapters`.
- Chapter: canonical `chapter` label, ordered `paragraphs`, and chapter-level
  `grammar` list.
- Paragraph: ordered `sentences`.
- Sentence: Japanese source text `jp`, target-language translation `en`,
  ordered tokenizer output `tokens`, and sentence-anchored `grammar`.
- Token: every token has surface text `t`. Kanji-bearing tokens may include
  hiragana ruby text `rt`. Content-word tokens with a gloss also include lexeme
  surface `w`, meaning `m`, note `n`, coarse POS `pos`, stable lexeme `key`
  `(lemma, lForm, pos1)` joined with U+001F, and first-occurrence boolean `new`.
- Grammar item: `pattern`, `explanation`, `example_jp`, and `s`, where `s` is
  the 1-based sentence number within the chapter or `0` for a general note.

An empty vault starts with neutral `general/knowledge` and `concept` fallbacks.
Ingestion extends Domain/Form only when a real page cannot be classified
accurately with the current taxonomy.

## 2. Frontmatter contract (content pages only)

This contract applies to **content pages under `wiki/entities/` and
`wiki/topics/`**. MOC files under `wiki/_index/` follow the
canonical-frontmatter rule in §1 and are exempt from this section.

Every content page starts with:

```yaml
---
type: Entity | Topic
page_id: 01KQ6MTGFP31DK5QDGBVHW3WJM   # immutable, rename-safe identity
aliases: [Nick Lane, 尼克·莱恩]    # source-language names + canonical
sources: [src_01abc, src_02xyz]    # source_ids cited by this page
last_ingested: 2026-04-22
---
```

- `sources:`, `aliases:`, and `page_id:` are maintained by `ingest.py`.
  Do not hand-edit.
- `page_id` is a 26-char ULID assigned once when the page is first
  created and never changed — even on rename. Tools (rewire, alias
  index, lint) reference pages by `page_id` so file renames don't rot
  incoming pointers. The LLM does **not** generate it; tooling injects
  it post-apply via `scripts/add-page-id.py`.
- `aliases:` always includes the canonical (English) title and every
  source-language name encountered. Deduplicate; no empty entries.

## 3. Zone markers + llm-zone format

Every wiki page has two zones:

```markdown
<!-- human-zone -->
Your own notes. LLM never touches this.
<!-- /human-zone -->

<!-- llm-zone -->
> [!AI] LLM Synthesis
>
> ### Optional sub-heading (use when the page has ≥ 3 claim clusters)
>
> A claim paragraph. Each paragraph is one self-contained thought and
> ends with at least one `[src:<source_id>]` citation. Markdown
> emphasis is fine: **bold**, *italic*, `code`.
>
> Another claim paragraph, separated from the previous one by a
> **blank `>` line** so Obsidian renders it as its own paragraph
> [src:<source_id>].
>
> - Bulleted lists allowed when enumerating discrete items; each
>   bullet ends with `[src:<id>]` [src:<source_id>].
> - `[[Wikilinks]]` to related entities are encouraged
>   [src:<source_id>].
<!-- /llm-zone -->
```

Formatting rules inside `llm-zone`:

- **Readability first.** One claim per paragraph. Always insert a
  blank `>` line between paragraphs so Obsidian renders them
  separately. Do **not** put every claim on its own line inside one
  paragraph — that renders as a wall of text.
- **Every paragraph (or bullet) ends with ≥ 1 `[src:<id>]`.**
- **Subheadings `> ###` are optional** — use only when the page has
  ≥ 3 thematic clusters.
- **Markdown features allowed**: `**bold**`, `*italic*`, `code`,
  bullet/numbered lists, `[[wikilinks]]`, short block-quotes for
  source-language fragments.

### Two-tier format — required when the page has ≥ 2 distinct sources

Once a page cites two or more distinct `source_id`s (counting the
ingest that's about to happen), the `llm-zone` body must be structured
as a rolling **Synthesis** section followed by append-only per-source
**Evidence** sections:

```markdown
<!-- llm-zone -->
> [!AI] LLM Synthesis
>
> ### Synthesis
>
> An integrating claim that draws on more than one source
> [src:a#§1, src:b#第二章].
>
> A second integrating paragraph [src:a#§2, src:c].
>
> ### From src:a#§1
>
> What this source-section contributed: detail, mechanism, or
> evidence specific to it [src:a#§1].
>
> ### From src:b#第二章
>
> What that chapter contributed [src:b#第二章].
<!-- /llm-zone -->
```

Rules:

- **`### Synthesis` is rewritten on each ingest.** It's a rolling
  multi-source summary; the LLM may compact older synthesis to keep it
  bounded (target: 3–6 paragraphs).
- **`### From src:<id>#<label>` sections are append-only.** One section
  per `<id>#<label>` actually ingested (whole-asset citations use
  `### From src:<id>`). When re-ingesting the same `<id>#<label>`,
  edit that section in place — do not duplicate or stack.
- **Citations inside Synthesis** may combine sources:
  `[src:a#§1, src:b#第二章]`. Single-source citations are fine too.
- **Citations inside a per-source Evidence section** cite only that
  source. Don't inject other sources' claims into another's Evidence
  block.
- **The heading `### From src:<id>#<label>` is NOT a citation.** It
  labels the section. Tools count only bracketed `[src:...]` as
  citations; never put bracket characters in the heading.
- **Single-source pages may stay in the simple one-block form.** They
  must convert to two-tier on the ingest that adds a second source:
  move the existing prose (which all cites the original source) into
  its `### From src:<orig_id>#<label>` section, write a fresh
  `### Synthesis` integrating both sources, and add the new
  `### From src:<new_id>#<label>` section with this run's evidence.

The prose-style and voice rules below apply unchanged inside both
Synthesis and per-source sections.

**Prose style — write for a human reader, not a search index.**

- Each paragraph is **2–4 sentences that build one idea together**,
  not a single stuffed sentence cramming every fact together with
  commas. Lead with the headline; follow with mechanism, evidence,
  or context that the reader needs to actually understand it.
- **Plain language first, jargon when earned.** Introduce a technical
  term with a brief plain gloss the first time you use it, then use
  it freely: `质子驱动力 (跨膜质子梯度)`, `NUMTs (核基因组中嵌入的
  线粒体DNA片段)`.
- **Prefer active voice** over passive when the actor is known:
  `米切尔于1961年提出化学渗透假说…` rather than
  `化学渗透假说于1961年被米切尔提出…`.
- Don't fragment a single idea across multiple paragraphs just to
  satisfy "one sentence per paragraph" — that's the opposite of the
  rule. Keep mechanism + evidence + nuance **together** in the
  paragraph they belong to.

Bad (dense, hard to parse):
> 线粒体通常被解释为源自α-变形菌内共生事件，其双层膜、独立DNA与核糖体等
> 特征支持其细菌起源 [src:X#第一章].

Good (narrative paragraph):
> 线粒体的起源通常被解释为约20亿年前的一次α-变形菌内共生事件：一个细菌
> 被宿主细胞吞入后没有被消化，反而长期共生下来。支持这种解释的特征包括
> 双层膜（外膜来自宿主，内膜来自细菌本体）、独立于核基因组的环状DNA，
> 以及与细菌高度相似的核糖体 [src:X#第一章].

**Voice — depends on page type.**

- **Entity pages (`type: Entity`) are encyclopedia-style.** Write
  declarative claims about the entity itself, not about how a
  source describes it. The `[src:<id>]` citation at the end of each
  paragraph already carries provenance — do **not** also preface
  claims with "the source claims", "this chapter argues",
  "本章主张", "文中指出" etc. on entity pages.

  Forbidden on entity pages (English): _the chapter / the section /
  the source / the text / the book / the author / according to /
  it is argued that_.

  Forbidden on entity pages (Chinese): _本章 / 本节 / 本书 / 该章 /
  该节 / 第N章 / 文中 / 书中 / 文献 / 作者认为 / 作者指出_.

- **Topic pages (`type: Topic`) are discussions** of how different
  sources frame a subject. Source-narrating prose is allowed but is
  **reserved for comparing distinct authorial positions across
  sources**, not as a generic way to phrase every claim.

  - **Default to declarative.** The `[src:<id>#<label>]` citation
    already records who wrote where; wrap a claim in
    source-narrating only when there's a genuine attribution to
    make.
  - **When you do attribute, name the AUTHOR and BOOK** (or
    hypothesis/theory). Don't use a chapter number as the
    grammatical subject — that omits whose claim it is, and the
    citation already carries the chapter. Form: `<author> 在
    《<book>》第N章中认为…` or `<hypothesis>主张…`.
  - **Distinguish authorial claims from relayed knowledge.**
    Attribute only when the source advances its own thesis. When
    the source merely cites well-established facts, history, or
    third-party data, present the claim declaratively — the source
    is your provenance, not the originator. Default to declarative
    when uncertain.

  Each comparative claim still ends with `[src:<id>]` so
  attribution is unambiguous.

**Uncertainty markers — both page types.** When a source's claim
is hedged, contested, or speculative, hedge in prose too: use
`may`, `might`, `likely`, `is proposed to`, `arguably`, `it
remains debated whether`, or in Chinese `可能`, `也许`, `据认为`,
`有人主张`, `仍有争议`, etc. The citation still goes at the end.
Do not present uncertain claims as if they were settled.

Examples:

Entity page (declarative):
> 线粒体获得是复杂生命出现的前提 [src:x#第一章]
> Mitochondria predate the last eukaryotic common ancestor
> [src:x#§3]

Topic page (declarative — preferred default):
> 金融化、货币政策与贫富差距可处于同一条链条上 [src:x#第7章].

Topic page (source-narrating, comparing across sources):
> 氢假说主张宿主是产甲烷古菌 [src:a#第一章]，而吞噬假说则认为宿主
> 是有原始吞噬能力的早期真核样细胞 [src:b].

Topic page (single-book authorial framing — name author + book):
> 付鹏在《见证逆潮》第7章中把利率病归因于危机应对的路径依赖
> [src:x#第7章].

Bad — chapter as agent (no author, redundant with citation):
> 第7章把利率病归因于危机应对的路径依赖 [src:x#第7章].

Bad — mis-attributes textbook history to the source's author:
> 付鹏指出美元脱钩黄金后形成石油美元体系 [src:x#第8章].

Good — same fact, declarative (the source is provenance, not originator):
> 美元脱钩黄金后形成石油美元体系 [src:x#第8章].

Uncertainty (either page type):
> 线粒体的内化事件可能发生在约20亿年前，但精确年代仍有争议
> [src:x#第一章].

The LLM may only edit text between `<!-- llm-zone -->` and
`<!-- /llm-zone -->`. Anything outside those markers is off-limits.

Obsidian renders `> [!AI]` as a callout, making LLM-authored content
visually distinct from hand notes.

## 4. Citation format

Every claim in the `llm-zone` ends with an inline citation:

```markdown
OpenAI released GPT-4 in March 2023 [src:01HPSRE42].
Inference cost dropped ~10x since launch [src:01HPSRE42,src:01HQAB73].
```

Multiple citations: comma-separated inside one pair of brackets.

`[src:<id>]` resolves only to a `source_id` declared in a sidecar
file under `sources/`.

### Section anchors

For long sources where only part of the asset was ingested (a
chapter of a book, a section of a long article), append the canonical
`#sec=` anchor followed by the RFC 3986 percent-encoded UTF-8 section label:

```markdown
线粒体获得是复杂生命出现的前提 [src:01KPWHET#sec=%E7%AC%AC%E4%B8%80%E7%AB%A0].
The comparison is bounded to one section [src:01HQAB73#sec=A%2C%20B%5D%20%23100%25].
```

Encoding is mandatory for newly emitted section anchors. It makes commas,
`]`, `#`, `%`, whitespace, and Unicode round-trip as one citation even inside
a comma-separated citation group. Producers MUST encode the label exactly
once; consumers split the group first and decode only the value after `#sec=`.
Legacy human-readable anchors such as `[src:<id>#第一章]` remain valid and MUST
continue to resolve. Structured media anchors such as `#card-2`, `#frame-3`,
and time ranges are not section labels and remain unencoded.

The anchor scopes the citation to the part of the source that was actually
ingested. When `ingest.py` is run with `--section-label`, every citation it
emits in that run carries the encoded anchor; without it, the citation is bare
`[src:<id>]` (whole-asset). The `<id>` portion still resolves to the same
sidecar — the anchor is informational provenance, not a separate source.

#### Structured media anchors

Media sources also have **structured** anchor namespaces, each valid ONLY
against a source of the matching capability:

| Anchor form | Grammar | Valid against |
|---|---|---|
| time range | `#H:MM:SS-H:MM:SS` | a transcript (video/audio/podcast) |
| card | `#card-N` (1-based) | an image_note |
| frame | `#frame-N` (1-based) | a frames source |

These three forms are mutually **disjoint** (the time-range regex requires a
`-`-separated pair of timecodes, so `#card-3`/`#frame-3` can't match it). The
media-anchor validators **reject** an anchor that matches one of these forms but
is cited against the wrong capability (e.g. `#frame-1` on a plain-video source).
This rejection is scoped to these structured namespaces ONLY — the free-form
human section anchors above (`#§3`, `#第一章`, `#<chapter-label>`) are never
grammar-checked and stay valid for text/book sources. `card-N`/`frame-N` resolve
against `.cards.json`/`frames.json` (a real card heading / a manifest ordinal),
not merely `1 ≤ N ≤ count`.

## 5. Conflict handling

When ingest finds a new claim that contradicts an existing one, the
LLM inserts an inline highlight on the affected page:

```markdown
==CONFLICT: src_02xyz claims X; existing from src_01abc says Y.==
```

Keep ingesting the rest of the source. Conflicts are resolved while
reading the page, not in a separate log.

Find all open conflicts: `rg "==CONFLICT:" wiki/`

## 6. Source-of-truth rule (anti-recursion)

**Only files in `sources/` are authoritative evidence.**

- Wiki pages are summaries, never sources.
- `[src:<id>]` must resolve to a `source_id` declared in a sidecar,
  never a wiki page title or filename.
- The LLM must never cite another wiki page as a source.

## 7. Source sidecars

Every asset in `sources/` has a companion `<asset>.md` sidecar. The
frontmatter is the authoritative metadata.

```yaml
---
source_id: 01HPSRE42ABCD        # ULID, immutable
type: source
sha256: e3b0c442...             # of the CANONICAL CITED ARTIFACT (see §7.1)
added: 2026-04-22T14:30:00Z
origin_type: file               # file | url | video | audio | image_note
origin_ref: "https://..."       # filename or URL
supersedes: null                # or [[<prior_source_id>]] wikilink
title: "Short human title"
author: "Author Name"            # optional; read from document metadata
---
```

Example: `sources/2026-04-22-karpathy-gist.pdf` →
`sources/2026-04-22-karpathy-gist.pdf.md`.

### 7.1 Media sources (`origin_type: video | audio | image_note`)

For document/URL sources the cited artifact *is* the stored file, so
top-level `sha256` hashes that file directly. **Media is different: what
`[src:<id>#…]` resolves against is a derived text artifact** (a
transcript, or OCR'd card text), not the raw media bytes. So for media:

- The **canonical cited artifact is committed and named so the sidecar
  is `<artifact>.md`** — i.e. the artifact is
  `sources/<slug>.transcript.md` (video/audio) or
  `sources/<slug>.cards.md` (image notes), and its sidecar is
  `sources/<slug>.transcript.md.md` / `sources/<slug>.cards.md.md`.
  **This double-`.md` is required, not a typo:** `lint.py`'s
  `all_sidecars()` / `check_source_drift` derive the asset as
  `sidecar.with_suffix("")`, so the sidecar must be the artifact name +
  `.md`. Naming the sidecar `<slug>.transcript.md` instead would make
  lint look for `sources/<slug>.transcript` (which doesn't exist) and
  **silently skip drift detection** for that source. **Top-level
  `sha256` hashes the artifact** — keeping the weekly re-hash and
  `check_source_drift` watching the bytes citations resolve against.
- The raw media file (if downloaded) is a **non-authoritative cache** at
  `sources/.media/<media_sha>.<ext>`. **Tracking rule:** the committed
  transcript/card artifact + its sidecar ARE tracked; `sources/.media/`
  is NOT. (Today the whole vault is local-only — `.gitignore` blankets
  `sources/` — so `.media/` is already covered; this rule is what to
  enforce once sources are pushed to a remote.)

A second committed artifact `sources/<slug>.transcript.json` (the structured
segments: per-segment `start`/`end`/`speaker`/`text`) is the re-render source and
the timestamp-anchor validation input; it is **hash-guarded** by
`media.transcript_json_sha256` (lint must re-hash it — it is not `*.md`, so the
generic `check_source_drift` won't catch it).

Media sidecars carry an extra `media:` block. **Video/audio** (transcript) —
fields are recorded **from the ASR service's `meta` payload, never hardcoded**;
a value the service does not (yet) emit is `null`/absent, never guessed:

```yaml
origin_type: video                 # video | audio
sha256: <sha of the committed `<slug>.transcript.md`>   # the cited artifact
media:
  platform: youtube                # youtube | podcast | rednote (platform ≠ origin_type)
  video_id: ba-HMvDn_vU            # YouTube canonical 11-char id — the DEDUP IDENTITY
  canonical_url: 'https://www.youtube.com/watch?v=ba-HMvDn_vU'
  service_source: '<canonical url submitted to the service>'   # = meta.source
  source_sha256: null              # raw-media hash; null when no media file is held locally
  source_bytes_stored: false
  upstream_drift_guard: none       # snapshot drift IS guarded (sha256 + transcript_json_sha256)
  speech_duration_s: 4231          # Σ(seg.end−seg.start) from the JSON — coverage-gate denominator
  duration_s: null                 # total MEDIA duration (⧗ download-recipe enhancement)
  transcript_kind: asr             # asr | subtitles | human
  transcript_tool: transcript-remote
  asr_engine: whisperx             # @version is ⧗
  asr_model: large-v3              # = meta.model
  device: cuda                     # = meta.device
  compute_type: float16            # = meta.compute_type
  diarize_requested: true          # = meta.diarized (the REQUESTED flag, not "≥2 speakers found")
  language: en                     # = top-level `language`
  # ⧗ recorded only once the service emits them (else null/absent — never guessed):
  align_requested: null
  align_succeeded: null
  diarize_succeeded: null
  transcript_job_id: null
  transcript_json_sha256: <sha of `<slug>.transcript.json`>
  render_format_version: 1
  transcribed: 2026-06-01T00:00:00Z
```

**Image notes** (`origin_type: image_note`, canonical artifact
`<slug>.cards.md`) use an OCR-shaped block instead:

```yaml
origin_type: image_note
sha256: <sha of the committed <slug>.cards.md>
media:
  platform: rednote            # a real platform, OR `unknown` (a RECOGNIZED value) for a
                               # manual export with no upstream id — never the literal `manual_export`
  post_id: 64f...              # platform post/note id or URL; ABSENT on a manual export
  identity_basis: image_post_id   # image_post_id (known post) | image_bundle (manual export)
  card_count: 9                # from the service meta — never counted locally
  # image_bundle_sha256 = sha256 over the index-ordered per-image image_sha256 values
  # (\n-joined, UTF-8, LF, NO trailing newline). On the image_bundle path it IS the
  # identity, so it is REQUIRED there (not "if downloaded"); on a known post it is
  # guarded *evidence/version* metadata, NOT part of the head key.
  image_bundle_sha256: <sha…>
  ocr_tool: paddleocr@2.7
  ocr_model: ch_PP-OCRv4
  ocr_params: '--lang ch'      # incl. resize/det_limit_side_len/angle-cls/preprocessing recipe
  ocred: 2026-05-28T12:00:00Z
  evidence_artifacts:          # §7.2 — SINGLE source of truth for non-canonical evidence hashes
    - {role: cards_json, path: sources/<slug>.cards.json, sha256: <…>}
    - {role: manifest,   path: sources/<slug>.cards.md.assets/_manifest.md, sha256: <…post-caption…>}
    - {role: image_bundle, sha256: <image_bundle_sha256>,
       bundle_recipe: 'sha256 of \n-joined index-ordered cards.json[].image_sha256'}
```

`<slug>.cards.json` is the **ordered per-card audit artifact** — each entry
`{index, heading_anchor, text, image_sha256, image_path}` (0-based `index`;
`heading_anchor` = the 1-based `card-N`; `image_path` required) — binding
order ↔ heading ↔ text ↔ image so swapped OCR text cannot pass. The
`image_bundle_sha256` is recomputed from these entries' `image_sha256` in `index`
order (lint recipe `image_sha256_index_join`).

**Podcasts** (`origin_type: audio`, `platform: podcast`) reuse the transcript
recipe block plus RSS-identity fields (`published` is podcast-only):

```yaml
media:
  platform: podcast
  feed_url: 'https://…/feed.xml'   # canonicalized per the dedup rules below
  episode_guid: '…'                # producer <guid>; may be absent → identity_basis cascade
  enclosure_url: '…'               # observation (tracking-wrapped/redirected)
  published: 2026-05-01            # podcast-only
  identity_basis: feed_guid        # feed_guid | feed_enclosure | feed_title_published
  resolution_source: feed_parse    # feed_parse | yt-dlp_info_json | user_supplied
  # …transcript recipe fields (transcript_kind/asr_*/transcript_json_sha256/…) as above…
```

**Video frames** (a frames re-ingest) produce a NEW source that `supersedes:`
the transcript source, **keeping the `.transcript.md` canonical-asset name** (so
the generic drift check still fires); the `.transcript.md`/`.transcript.json` are
carried forward byte-for-byte (`transcript_json_sha256` unchanged — visual-only,
not a retranscription). The manifest `<slug>.transcript.md.assets/frames.json`
pins a 1-based ordinal `N` → `{timecode, filename, ocr_text}`; both `frames.json`
and the frame bundle (sha256 over index-ordered per-frame `image_sha256`) appear
in `media.evidence_artifacts[]`.

### 7.2 Generalized media identity, aliases & evidence artifacts

These fields generalize the `media:` block across all kinds:

- **`media.identity_basis`** — names which tuple is the dedup key (enum:
  `youtube_video_id | image_post_id | feed_guid | feed_enclosure |
  feed_title_published | image_bundle`). The shared head-resolver dispatches on
  `media.platform` then compares **like-basis to like-basis**; an unrecognized
  basis ⇒ hard error. **Legacy compat:** an existing `platform: youtube` sidecar
  with NO `identity_basis` is read as `youtube_video_id` (so the strict resolver
  never bricks committed Phase-2 sources).
- **`media.identity_aliases[]`** — basis-qualified `{basis, key}` entries.
  When a later, *stronger* basis resolves for an episode already known under a
  weaker one (e.g. an enclosure-keyed podcast later gets a real `episode_guid`),
  the **default is a hard error** (operator reconciles via an explicit command,
  which records the alias) — never silent auto-supersede.
- **`media.evidence_artifacts[]`** — `{role, path, sha256, bundle_recipe?}`, the
  **single source of truth** for every committed NON-canonical evidence artifact
  (`.transcript.json`/`.cards.json`/`_manifest.md`/`frames.json`/image &
  frame bundles). Lint re-hashes each; do NOT also carry scalar `*_sha256` fields
  that could disagree. The canonical cited artifact (`.transcript.md`/`.cards.md`)
  is governed by the top-level `sha256` and is NOT listed here (no double hash).
  Paths are repo-relative, under `sources/`, git-tracked, no `..`. **Legacy
  compat:** a transcript sidecar with no `evidence_artifacts[]` maps to the
  legacy `{canonical .md (top-level sha256), .transcript.json
  (transcript_json_sha256)}` pair.
  **Transition status:** until the lint generalization that *reads*
  `evidence_artifacts[]` lands (plan §8.4), the scalar `transcript_json_sha256`
  remains the ACTIVE drift guard and is the field new media sidecars carry; they do
  NOT yet emit `evidence_artifacts[]`. Both land together in §8.4, at which point
  lint reads the list and the scalar becomes derived/legacy — so "single source of
  truth" is the post-§8.4 end state, not a contradiction with today's sidecars.

**Dedup identity** keys on an **evidence bundle**, NOT the artifact sha
alone (two different videos can share a transcript). Required bundle
fields by kind:

- YouTube video/audio (`identity_basis: youtube_video_id`): the **normalized
  `media.video_id`** IS the dedup identity (the canonical 11-char id; see
  `plan/expansion-plan.md` §7.2 for the URL-normalization rule). `origin_ref`
  (raw user URL) and `canonical_url` are retained for audit but are **not** part
  of the key; `source_sha256` is absent on the remote-ASR path. Re-ingest scans
  tracked media sidecars for a matching `media.video_id` and resolves the **head**
  of the `supersedes:` chain; an ambiguous/cross-identity/unparseable chain is a
  **hard error**, never a guess.
- Podcast: `feed_url` + `episode_guid` (basis `feed_guid`), cascading to
  `feed_url` + `enclosure_url` (`feed_enclosure`) then `feed_url` + `title` +
  `published` (`feed_title_published`) when the GUID is absent. `feed_url` is
  canonicalized (lower-case scheme+host, strip leading `www.`, strip a default
  port (80/443), drop fragment, collapse trailing slash; path/query left
  byte-as-is — NOT %-decoded, since decoding reserved delimiters could merge
  distinct URLs) and reconciles as hard as the GUID.
- Image note: known post = `platform` + `post_id` (basis `image_post_id`);
  `image_bundle_sha256` is guarded evidence, NOT a head key — a known post whose
  bundle changed is a **hard error** unless `--reocr`/supersede is explicit.
  Manual export (no `post_id`) = `platform` (`unknown`/real) + `image_bundle_sha256`
  (basis `image_bundle`).

(These identity fields are all keys of the `media:` block.)

## 8. Source versioning

`source_id`s are immutable.

- If a source's content changes, **do not overwrite** it.
- Create a new sidecar with a new `source_id`.
- Set `supersedes: [[<old_id>]]` on the new sidecar.
- Re-ingest under the new id.

This keeps all citations in the wiki valid forever: `[src:<old_id>]`
still resolves, and the supersession wikilink tells you the chain.

**Media (§7.1):** the committed transcript/OCR artifact is immutable
like any source — never rewrite it in place. Re-transcribing with a
better ASR model/params produces different text, so the artifact's
top-level `sha256` changes and it gets a **new `source_id` with
`supersedes:`**, even when the raw media bytes are unchanged — a clean
supersession, not a silent edit. (This is the *content/version* hash;
it is distinct from the *dedup identity* — the evidence bundle in §7.1 —
which is what decides "have I already ingested this source?".) Mutable
upstream captions (YouTube auto-captions) are out of scope unless
deliberately refreshed (re-fetch → compare artifact hash → supersede on
change).

**Phase-3 supersession rules.** Re-OCR of an image_note (`--reocr`) = a new
`source_id` with `supersedes:` (the card text *is* the OCR output, so there is
**no** card `render_format_version`). Adding video
**frames** to an existing transcript = a new source that `supersedes:` the
transcript, carries it byte-for-byte, and rewrites live `[src:<old>#…]` citations
to the new head (then re-runs the timestamp gate). A frame asset's `_manifest.md`
is **frozen** after the post-commit caption pass writes `manifest_sha256`; a later
recaption is itself a supersede, not an in-place edit.

## 9. Weak-evidence rule

If a question cannot be answered from cited sources, the wiki answers
`unknown` or `uncertain`. Never synthesize confident claims without
provenance.

## 10. Naming

- **Wiki filenames match the page language.**
  - English pages: kebab-case ASCII (`nick-lane.md`,
    `mitochondria.md`).
  - Chinese pages: the Chinese term, no spaces (`线粒体.md`,
    `氢假说.md`). Obsidian and git handle non-ASCII fine.
- **Page `# H1`** matches the filename (same language, same wording).
- **`aliases:`** lists every other surface form (other language
  translations, transliterations, variants) so Obsidian resolves
  `[[Nick Lane]]`, `[[尼克·莱恩]]`, and `[[nick-lane]]` all to the
  same file.
- **Wikilinks in prose**: use the target page's own title — e.g.,
  link to `[[Mitochondria]]` if the page is `mitochondria.md`, or
  `[[线粒体]]` if the page is `线粒体.md`. From a different-language
  source you may still use the target page's native title; Obsidian
  resolves via aliases.
- Source sidecars: `<original-asset-filename>.md`.

## 11. Multi-language handling

All `llm-zone` prose is in the page's own language (§0: source
language wins at page creation, then stays).

- `aliases:` must include English + Chinese forms (and any other
  source-language surface form seen) for every page.
- First mention in prose, when the page's language differs from the
  current source's language, shows the other form in parens:
  - Chinese page, English source: `线粒体 (Mitochondria)`.
  - English page, Chinese source: `Mitochondria (线粒体)`.
- Subsequent mentions use the page-language form only.
- For semantically important phrasings, quote the source-language
  fragment inline: `> "线粒体是十亿年前一场交易的遗产" [src:<id>]`.

## 12. Things that never happen

- LLM editing outside `llm-zone`.
- LLM citing a wiki page as a source.
- Hand-editing a source sidecar.
- Overwriting an asset or sidecar when content changes (use
  supersession instead).
- Adding a new `metadata.json` / `manifest.json` / `state.json`
  without updating this schema first.
- LLM using a tag that is neither already in `wiki/_taxonomy.md` nor appended
  there in the same diff (see §13).

## 13. Tags

Content pages under `wiki/entities/` and `wiki/topics/` carry a
`tags:` field for categorization. MOC files in `wiki/_index/` are
generated from the union of these tags.

```yaml
tags: [biology/cell, mechanism]
```

**Source of truth**: `wiki/_taxonomy.md`. Ingestion reuses existing tags when
they fit and may append the minimum missing Domain/Form tags in the same diff
as pages that use them. Existing taxonomy lines are immutable: ingestion may
not delete, rename, reorder, or rewrite them.

**Tag syntax** (regex, lint-enforced):

```
^[a-z0-9]+([-/][a-z0-9]+)*$
```

- Lowercase ASCII letters, digits, `/`, and `-` only. **No
  underscores** (the slug rule depends on `_` being absent).
- `/` separates hierarchy levels (e.g., `biology/cell`); `-`
  separates words within a level (`history-of-science`).
- No leading/trailing/adjacent separators.
- Hierarchy depth ≤ 2 (one `/` max).

**Cardinality** (lint-enforced):

- Exactly **1 Form** tag (e.g., `concept`, `mechanism`).
- Exactly **1 primary Domain** tag (e.g., `biology/cell`); the
  first Domain tag in list order is the primary.
- Optional **0–2 secondary** Domain or Reserved tags.
- Total: 2–4 tags.

**YAML format constraint**: emit `tags:` in single-line flow
(`tags: [a, b, c]`) or block style. Multi-line flow
(`tags: [\n  a,\n  b\n]`) is rejected by lint and would silently
corrupt under surgical line-edit tooling.

**Reserved tag**: `taxonomy-gap` — page legitimately needs a
Domain not yet in the taxonomy. Add **alongside** the closest
available Domain tag (it does NOT satisfy the "1 primary Domain"
requirement on its own).

Tags are an indexing layer, not content. Page bodies remain in
their source language; tags stay ASCII regardless.

## 14. Images

Images extracted from sources (EPUB/MOBI embedded images, PDF figures, web
HTML `<img>` URLs) live in **`sources/<asset>.assets/<sha12>.<ext>`**.
A frontmatter-only YAML manifest at
`sources/<asset>.assets/_manifest.md` records each image's sha256,
size, dimensions, origin_refs (chapter/page/URL provenance),
`decorative` flag, and `caption` (filled in by `scripts/caption.py`
via a vision-capable CLI). See `plan/image-ingest-plan.md` §3–§5 for
naming + extractor details.

MOBI/AZW sources are converted through KindleUnpack before extraction; their
assets still live beside the original `.mobi`/`.azw*` file.

**LLM-emitted embeds in wiki pages**:

- Use **Obsidian transclude syntax** `![[<path>]]` or
  `![[<path>|alias]]`. Standard markdown image syntax `![alt](path)`
  is forbidden inside `<!-- llm-zone -->` (lint #19).
- Embed paths point at `sources/<asset>.assets/<file>` (lint #14).
- External URLs `http(s)://...` are forbidden in any embed inside
  llm-zone (lint #18). All images must be cached locally first.
- Images flagged `decorative: true` in the manifest must NOT be
  embedded (lint #16).

**Where embeds may appear** (lint #15):

| Page state                       | Embed location                                                              |
|----------------------------------|-----------------------------------------------------------------------------|
| Single-source page (1 cited src) | Inside the page's only `> [!AI]` callout                                    |
| Multi-source page (≥ 2 srcs)     | Inside `### From src:<id>#<label>` Evidence sections; asset dir must match the cited `src:<id>` |
| `### Synthesis` block            | Forbidden — synthesis is rolling; embeds belong with the Evidence they support |
| `<!-- human-zone -->`            | Allowed and unchecked — user-owned                                          |

**Page transcludes** (e.g. `![[Some Other Page]]` with no image
extension) are forbidden inside llm-zone (lint #20). Wiki pages link
to other pages via plain `[[wikilinks]]`, not transcludes.
